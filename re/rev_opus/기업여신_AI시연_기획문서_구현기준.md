# 기업여신 AI Agent 시연 — 기획문서 (구현 기준)

> 본 문서는 `new/html/p1.html`~`p12.html` 및 `js/common.js`의 **실제 구현 코드**를 근거로 역분석하여 작성되었다.  
> `new/docs` 하위의 RFP 요건 문서 및 설계문서는 보조 참고 자료로만 활용하였으며, 구현과 설계가 불일치하는 항목은 명시적으로 기술한다.  
> 향후 재구현·보강을 위한 관점을 포함한다.

---

## 1. 시연 개요

### 1.1 배경 및 목적

은행 기업여신(법인대출) 업무에서 고객 제출 서류의 접수·검증·심사 입력을 자동화하는 AI Agent의 역량을 제안 설명회용 시연 화면으로 보여주는 것이 목적이다. RFP가 요구하는 「여신 서류 입력 및 검수 자동화」 요건 중 핵심 E2E 흐름—서류 등록부터 심사 요청까지—을 12단계(P1~P12)로 구성하여 한 번의 자동 재생(auto-paging)으로 전체 프로세스를 시연할 수 있도록 한다.

### 1.2 핵심 메시지

| # | 메시지 | 시연에서의 대응 |
|---|--------|---------------|
| 1 | AI Agent가 서류 분류·전처리·OCR·검증을 자율 수행한다 | P3 → P6 → P9 Agent 화면 자동 진행 |
| 2 | RPA 연동으로 진위검증을 자동화한다 | P5 RPA 프로그레스 바 + 문서별 기관 API 호출 시연 |
| 3 | 사람(HITL)이 최종 확인·수정하며 책임 소재를 확보한다 | P8 값 수정 데모, P11 결과 확인 데모 |
| 4 | 오류 발생 시 실시간 알림으로 즉시 대응한다 | P7 브릿지 미니 오류 알림, P10 완료 알림 |
| 5 | 검증 완료 값은 통합단말로 자동 전달된다 | P12 심사정보 자동입력 화면 |

### 1.3 시연 대상 업무 범위

**포함 범위 (In-scope)**

- 기업여신 신청정보 입력
- 첨부 서류 등록 및 BPR 전송
- Agent 실행 계획 수립 (문서 분류·큐 생성·Task Graph)
- 문서 전처리 (기울기 보정, 노이즈 제거, 명암/대비 보정, 페이지 정렬)
- 진위검증 (RPA 연동, 기관 API 호출, 응답 파싱, 진위 비교)
- AI-OCR 및 필드 추출 (텍스트 인식, 필드 매핑, 신뢰도 평가)
- 오류/완료 알림 (브릿지 미니 UI)
- HITL 값 수정 및 결과 확인 (브릿지 화면)
- 대사/검증 (입력값 대사, 문서간 대사, 필수값 누락, 타입/유효값 검증, KYC 교차검증, 담보정보 교차검증)
- 담보물건 교차검증 (부동산등기부등본·건축물대장·사정의견서 기반 LTV 산출)
- 심사정보 자동입력 및 심사 요청 (본부 전결 판정 포함)

**미포함 범위 (Out-of-scope)**

- 원본 설계의 S6(전결권 상향 검증), S11(주주명부 자동입력), S14(전결권 선택/등록), S15(반송 확인) — 현재 시연에는 미구현
- S10(담보서류 검증)은 부동산등기부등본·건축물대장·사정의견서를 통한 담보정보 교차검증으로 반영
- 실제 백엔드 API 연동, 데이터 영속화
- 사용자 인증/인가, 세션 관리
- 반응형 레이아웃 (고정 1280×720 캔버스만 지원)

## 2. 사용자 역할 및 채널

### 2.1 역할 정의

| 역할 | 설명 | 시연 내 등장 |
|------|------|------------|
| 영업점 담당자 (RM) | 신청정보 입력, 서류 업로드, HITL 수정/확정, 심사 요청 | P1, P2, P7, P8, P10, P11, P12 |
| AI Agent | 문서 분류·전처리·OCR·검증 백그라운드 자동 수행 | P3, P4, P5, P6, P9 |

### 2.1-1 단말-Agent-브릿지 역할 분리 원칙

> **재설계 확정 보완서 §0 참조**

```
단말이 AI 호출 → Agent 수행 (필요시 HITL) → 브릿지 알림 → 브릿지에서 결과 확인 → 단말에 적용
```

- **단말 화면은 AI 결과를 직접 표시할 수 없다.** AI 처리 결과는 반드시 브릿지(HITL 검토)를 거쳐 WINI 전송 형태로 단말에 도달한다.
- P2의 "AI 검수 요청"은 AI를 호출하는 **트리거**이므로 원칙에 부합한다.
- P12의 재무/전결 데이터는 "브릿지 수신" 데이터로 표현한다.

### 2.2 채널 (화면 유형)

| 유형 | 구현 특성 | 해당 화면 |
|------|----------|----------|
| **단말** (Terminal) | 흰색 배경, 상단 `영업점 단말` 뱃지, 은행 단말 UI 톤 | P1, P2, P12 |
| **Agent** | 회색 배경(`#edf0f3`), 상단 `백그라운드 실행` 뱃지, 5단계 프로세스 플로우 바 고정, 좌/우 패널 | P3, P4, P5, P6, P9 |
| **브릿지** (Bridge) | 연두 배경(`#f1f8ea`), 상단 `AI 브릿지` 뱃지, 좌측 문서목록 + 우측 멀티뷰 패널 | P8, P11 |
| **브릿지 미니** (Bridge Mini) | 단말 화면 위 우하단 오버레이(370px 카드), 알림 슬라이드업 애니메이션 | P7, P10 |

## 3. E2E 시연 시나리오 (P1 → P12)

### 3.1 전체 흐름도

```
P1(단말) → P2(단말) → P3(Agent) → P4(Agent) → P5(Agent) → P6(Agent)
     → P7(브릿지미니) → P8(브릿지) → P9(Agent) → P10(브릿지미니)
     → P11(브릿지) → P12(단말)
```

### 3.2 화면별 상세

#### P1. 신청정보 입력 — 단말

- **목적**: 영업점 담당자가 기업여신 신청 기본정보를 입력하는 장면 시연
- **주요 UI**: 2열 폼 그리드, 영업점 코드/담당자 팝업 조회(모달), select 드롭다운 커스텀 메뉴
- **시연 데모**: 자동 타이핑으로 차주명(`(주)000바이오`), 신청번호, 영업점(411/강남기업금융센터), 담당자(김담당), 대출상품(기업부동산담보대출), 금액(30억), 만기일, 상환방식(원리금균등), 취급구분(신규), 자금용도(공장부지 매입), 담보유형(부동산담보), 신청채널(영업점) 순차 입력
- **데모 데이터**: `demoFields` 객체에 하드코딩
- **전환**: 모든 필드 입력 완료 후 `첨부문서 등록` 버튼 클릭 → 자동 페이지 이동

#### P2. 첨부문서 등록 — 단말

- **목적**: 서류 업로드 및 BPR 전송 시연
- **주요 UI**: 상단 메타(신청번호, 차주명 pill), 서류 목록 테이블(체크박스, 문서구분, 문서명, 파일명, 등록일시, 검증상태), 파일 선택 모달(picker), BPR 전송 로딩/완료 모달
- **시연 데모**: 초기 9건 중 8건 미등록 → 파일 선택 팝업 전체 체크 → 확인 → 8건 등록완료 → BPR 전송 → 완료 팝업 → `서류입력 및 검수(AI)` 버튼 클릭
- **등록 서류**: 사업자등록증명원, 법인등기부등본, 표준재무제표증명, 부가가치세 과세표준증명원, 법인인감증명서 (기본 5건), 부동산등기부등본, 건축물대장, 사정의견서 (담보 3건), 주주명부 (조건부 1건/미등록)
- **전환**: 자동 페이지 이동

#### P3. 실행 계획 수립 — Agent

- **목적**: Agent가 문서를 분석하고 실행 계획을 수립하는 과정 시각화
- **주요 UI**: 프로세스 플로우 바(5단계, 현재 단계 녹색 펄스), 좌측 문서 분석 목록(8건), 우측 실행 계획(4개 대상 카드 + 실행 순서 타임라인)
- **시연 데모**: 문서 8건 순차 `구조 인식 완료` → 4개 카드에 대상 문서 채움(전처리/OCR/진위검증/대사 대상) → 4단계 실행 순서(문서 분류 → 큐 생성 → Task Graph → Worker 할당) 순차 완료
- **전환**: 약 10초 후 자동 이동

#### P4. 문서 전처리 — Agent

- **목적**: OCR 정확도 향상을 위한 문서 전처리 과정 시각화
- **주요 UI**: 플로우 바(문서 전처리 활성), 좌측 전처리 대상 문서 5건(기본 3건 + 담보 2건), 우측 전처리 4개 작업 카드(기울기 보정, 노이즈 제거, 명암/대비 보정, 페이지 정렬) + 실행 순서 타임라인
- **시연 데모**: 대상 문서 채움 → 4단계 순차 진행/완료 → 문서 뱃지 `전처리 완료`
- **전환**: 자동 이동

#### P5. 진위검증 — Agent

- **목적**: RPA 연동 진위검증 과정 시각화
- **주요 UI**: 플로우 바(진위검증 활성), RPA 프로그레스 바(비선형 증가), 검증 테이블(5건 문서 × 3단계: 기관 API 호출 → 응답전문 파싱 → 진위 확인)
- **대상 문서**: 사업자등록증명원(국세청), 법인등기부등본(대법원 등기소), 법인인감증명서(국세청), 부동산등기부등본(대법원 등기소), 건축물대장(국토부)
- **시연 데모**: RPA 로딩 0→100% → 5문서 병렬 파이프라인(API 호출 → 파싱 → 비교) 순차 완료
- **전환**: 자동 이동

#### P6. AI-OCR 및 필드 추출 — Agent

- **목적**: AI-OCR 엔진의 텍스트 인식·필드 매핑·신뢰도 평가 시각화
- **주요 UI**: 플로우 바(OCR 활성), 추출 테이블(8건 문서 × 3단계: 텍스트 인식 → 필드 매핑 → 신뢰도 평가), 하단 메시지 전송 Tool 프로그레스
- **핵심 시나리오**: 법인등기부등본(row 1)만 신뢰도 평가에서 `검토필요`(warn) — 나머지 7건은 `완료`(done)
- **시연 데모**: 8건 병렬 인식 → 매핑 → 신뢰도 → 법인등기부등본 검토필요 → 플로우 바 alert(빨간 깜빡임) → 메시지 전송 Tool 로딩 → 전송완료
- **전환**: 자동 이동

#### P7. 검토 요청 알림 — 단말 + 브릿지 미니

- **목적**: 오류 발생 시 영업점 단말 위에 실시간 알림이 뜨는 것을 시연
- **주요 UI**: 배경은 일반 은행 단말 화면(고객정보, 계좌개설, 거래내역 등), 우하단 브릿지 미니 카드(슬라이드업 0.8s)
- **알림 내용**: "법인등기부등본 신뢰도평가 검토필요 1건 발생. 값 확인 및 수정이 필요합니다."
- **뱃지**: 빨간색 `오류`
- **전환**: 4.2초 후 자동 이동

#### P8. 값 수정(HITL) — 브릿지

- **목적**: 담당자가 AI 추출 결과를 직접 검토·수정하는 HITL 과정 시연
- **주요 UI**: 상단 5단계 플로우 바(클릭 전환 가능), 좌측 문서 목록(8건, 상태 뱃지), 우측 멀티뷰 패널
  - **실행 계획 수립 뷰**: 문서 메타정보 + 대상 판단 플래그(전처리/OCR/진위검증/대사)
  - **문서 전처리 뷰**: 4개 전처리 작업 상태
  - **진위검증 뷰**: 발급기관, API 호출/파싱/진위확인 상태
  - **AI-OCR 뷰**: 좌측 문서 원본(paper-table), 우측 추출 항목 테이블(항목명/값/신뢰도/근거ID) + 값 수정 에디터
  - **대사/검증 뷰**: 플레이스홀더(구성 예정)
- **핵심 시나리오**: 법인등기부등본 선택 → 자본금 필드 선택 → OCR 값 `1,000,000,000원`을 `1,600,000,000원`으로 수정, 변경사유 `OCR 오인식(스캔 품질)` 입력 → 변경 적용
- **근거 하이라이트**: 좌측 원본 테이블에서 선택된 필드 노란색 하이라이트, evidence ID 표시
- **전환**: `작업 재개` 버튼 클릭 후 자동 이동

#### P9. 대사/검증 — Agent

- **목적**: 입력값과 추출값의 교차 검증 과정 시각화
- **주요 UI**: 플로우 바(대사/검증 활성), 검증 테이블(2건 문서 × 8단계: 입력값 대사 → 문서간 대사 → 필수값 누락 → 타입 검증 → 유효값 검증 → KYC 교차검증 → **담보정보 교차검증** → 최종 판정), 하단 메시지 전송 Tool
- **대상 문서**: 사업자등록증명원, 법인등기부등본 (기본 2건) + 부동산등기부등본, 건축물대장, 사정의견서 (담보정보 교차검증 대상 3건)
- **담보정보 교차검증**: 부동산등기부등본·건축물대장·사정의견서에서 추출한 소재지·면적·추정가액을 교차 대사하여 LTV(62.5%) 산출 결과를 검증
- **시연 데모**: 기본 2건 동시 시작(입력값 대사) → 각 문서 순차 진행 → 담보정보 교차검증 → 최종 판정 완료 → 메시지 전송
- **전환**: 자동 이동

#### P10. 검토 완료 알림 — 단말 + 브릿지 미니

- **목적**: Agent 전체 작업 완료 후 단말에 완료 알림 표시
- **주요 UI**: P7과 동일한 배경 단말 화면, 우하단 브릿지 미니
- **알림 내용**: "[신청번호 : CC411-2026-000123] 제출서류 AI검토가 완료되었습니다. 결과를 확인 해주세요."
- **뱃지**: 파란색 `완료`
- **전환**: 4.2초 후 자동 이동

#### P11. 결과 확인(HITL) — 브릿지

- **목적**: 담당자가 Agent 전체 처리 결과를 단계별로 최종 확인 (담보정보 교차검증 결과 포함)
- **주요 UI**: P8과 동일한 레이아웃 구조 (플로우 바, 문서 목록(8건), 멀티뷰 패널)
- **주요 차이점 (vs P8)**:
  - 모든 단계가 `done` 상태 (P8에서는 OCR 단계가 error)
  - 법인등기부등본 OCR 추출값에 자본금이 이미 `1,600,000,000원`으로 보정 완료, 신뢰도에 `입력` 표기
  - 변경사유 필드에 `OCR 오인식(스캔 품질)` 사전 입력
  - 하단 버튼이 `WINI 전송` (통합단말 전송 의미)
- **시연 데모**: 플로우 바 단계 순차 클릭(문서 전처리 → 진위검증 → OCR → 대사/검증) → WINI 전송 버튼
- **전환**: 자동 이동

#### P12. 심사정보 자동입력 및 심사 요청 — 단말

- **목적**: 브릿지(P11)에서 WINI 전송된 검증 완료 데이터를 단말에서 수신하여 심사를 요청하는 최종 단계
- **역할 분리 원칙**: 단말은 AI 결과를 직접 표시하지 않음. 재무 정보·담보 평가·전결 판단 등은 모두 "브릿지 수신" 데이터로 표현
- **주요 UI**: 3열 폼 그리드(25개 필드, 모두 readonly), "WINI 수신완료" 뱃지, 수신 상태 텍스트, 심사요청 버튼, 완료 모달
- **자동입력 필드**: P1 입력값 13건 + 담보 정보 5건(소재지/토지면적/건물면적/추정가액/LTV) + 브릿지 수신 재무정보 4건 + 전결 레벨 + 전결 사유 + 심사 메모 = 총 25건
- **섹션 구성**: 기본 정보 / 대출 정보 / **담보 정보 (브릿지 수신)** / **재무 정보 (브릿지 수신)** / **전결 / 심사 (브릿지 수신)**
- **전결 판정**: 본부 전결 (금액 기준 초과 + 부동산 담보 LTV 검토)
- **시연 데모**: "브릿지로부터 WINI 데이터 수신 중..." → 전체 필드 페이드인 → "브릿지 수신 완료" → 심사요청 클릭 → 완료 모달("본부 심사 단계로 이관")

## 8. 기획 관점 강점 및 한계

### 8.1 강점

1. **완결된 E2E 스토리텔링**: 12단계 자동 재생으로 서류 접수→AI 처리→사람 확인→심사 요청까지 끊김 없이 보여줌
2. **HITL 시나리오 구체성**: OCR 오인식이라는 현실적 오류를 선정하고, 값 수정+사유 입력+근거 하이라이트까지 일관되게 연출
3. **Agent-사람 경계 명확**: 화면 유형(단말/Agent/브릿지)과 뱃지로 "누가 하는 작업인지"를 시각적으로 즉시 구분
4. **검증 깊이**: P9에서 6종 검증(입력값 대사, 문서간 대사, 필수값 누락, 타입, 유효값, KYC 교차)을 모두 시연하여 전문성 표현

### 8.2 한계 및 보강 필요 사항

1. **데이터 하드코딩**: 모든 데모 데이터가 각 HTML 인라인 JS에 분산 하드코딩 → 공통 데이터 모듈 분리 필요
2. **상태 비공유**: 페이지 간 `location.href` 전환으로 상태 단절 → P8에서 수정한 값이 P11/P12에 자동 반영되는 것처럼 보이지만, 실제로는 각 페이지에 독립 데이터를 미리 세팅
3. **제한된 인터랙션**: auto-paging 모드에서는 사용자 개입 불가. 설명회에서 특정 화면 정지/되돌아가기 불편
4. **P7/P10 배경 화면**: 여신 업무와 무관한 일반 은행 단말(고객정보/계좌개설)이 배경 → 여신 업무 맥락의 단말로 교체 필요
5. **반응형 미지원**: 1280×720 고정 → 프로젝터/모니터 해상도에 따른 대응 부재
6. **코드 중복**: P8과 P11은 1,350줄 이상의 거의 동일한 코드 → 공통 컴포넌트 추출 필요
7. **P12 자동 재생 없음**: 시퀀스 마지막이므로 의도적이나, 심사요청 클릭까지 자동화할지 기획 결정 필요

| ~~담보서류 검증~~ | **반영 완료** — 담보 3종 서류 기반 교차검증 | ~~중간~~ |
| 주주명부 입력 | OCR 구조화 → CI101 자동반영 | 중간 |
| 약정서 검증 | 인감/서명 누락 감지, 특약 분석 | 낮음 |

### 9.2 기술 기반 개선 방향

| 항목 | 현재 | 개선 방향 |
|------|------|----------|
| 데이터 관리 | HTML 인라인 하드코딩 | 중앙 JSON/JS 모듈로 분리, 시나리오별 프리셋 |
| 상태 관리 | 없음 (페이지별 독립) | sessionStorage 또는 SPA 전환 + 글로벌 store |
| 코드 재사용 | 없음 (복사-붙여넣기) | 공통 컴포넌트(플로우 바, 문서 목록, 추출 테이블 등) 모듈화 |
| 시연 제어 | auto-paging ON/OFF만 가능 | 시연 대시보드(단계 점프, 속도 조절, 일시정지) 추가 |
| 스타일 | 각 HTML에 인라인 style | 공통 CSS 파일 + CSS 변수 체계화 |

### 9.3 시연 운용 관점

- **시연 모드**: 전체 자동 재생(현재) / 스텝 바이 스텝(수동) / 하이브리드(특정 구간 자동+설명 구간 정지)
- **녹화 대응**: 1280×720 고정 캔버스 유지, 애니메이션 0.2~0.8초 원칙 계속 적용
- **PPT 캡처 대응**: 각 화면의 최종 상태(모든 뱃지 완료) 스냅샷 자동 생성 기능 고려

## 부록 B. 용어 정의

| 용어 | 정의 |
|------|------|
| HITL | Human-in-the-Loop. AI 결과를 사람이 검토·수정하는 프로세스 |
| BPR | Business Process Reengineering. 여기서는 서류 문서를 W/F 시스템으로 전송하는 행위 |
| RPA | Robotic Process Automation. 발급기관 API를 자동으로 호출하는 소프트웨어 로봇 |
| WINI | 은행 통합단말(Window Integrated) 시스템 약칭 |
| 대사 | 두 값을 비교·대조하여 일치/불일치를 확인하는 행위 |
| 신뢰도 | OCR 엔진이 추출 결과에 부여하는 정확도 점수 (0~100%) |
| 근거 ID (Evidence ID) | 추출 필드와 원본 문서 위치를 매핑하는 고유 식별자 |
| 플로우 바 | Agent 화면 상단의 5단계 프로세스 진행 표시 UI |
| 브릿지 미니 | 단말 화면 우하단에 오버레이로 표시되는 소형 알림 카드 |
