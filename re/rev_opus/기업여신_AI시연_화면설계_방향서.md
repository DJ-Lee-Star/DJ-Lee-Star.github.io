# 기업여신 AI Agent 시연 — 화면 설계 방향서

> 본 문서는 우리은행 기업여신 AI Agent 시연 화면의 **디자인 방향성, 레이아웃 원칙, 시각 체계**를 정의한다.  
> 개별 화면(P1~P12)의 구체적 컴포넌트 배치가 아닌, **화면 유형별 설계 원칙과 디자인 시스템**을 다룬다.  
> 시연 환경(프로젝터, 대형 모니터, 녹화)의 특수성과 우리은행 CI를 반영한다.

---

## 1. 설계 철학

### 1.1 핵심 원칙

| 원칙 | 설명 | 시연에서의 의미 |
|------|------|---------------|
| **Glanceable** | 3m 거리에서도 즉시 파악 가능 | 프로젝터 시연 시 뒷줄 참석자도 핵심 정보 인지 |
| **Storytelling** | 화면 전환이 곧 서사 진행 | "누가(단말/Agent/브릿지), 무엇을, 왜"가 화면을 보는 즉시 드러남 |
| **Trust** | 은행 업무 시스템의 신뢰감 | 절제된 색상, 정돈된 그리드, 일관된 타이포 |
| **Clarity** | 화면 유형 간 명확한 시각 구분 | 배경색·헤더·뱃지만으로 현재 화면이 어디인지 즉시 인지 |
| **Consistency** | 동일 요소는 어디서나 동일하게 | 디자인 토큰 기반 일관성, "비슷한데 다른" 요소 제거 |

### 1.2 현행 디자인 문제점 진단

| # | 문제 | 현황 | 영향 |
|---|------|------|------|
| 1 | **디자인 토큰 부재** | 색상·간격·폰트를 각 HTML에 인라인 중복 정의 | 유사하지만 미세하게 다른 값들이 산재 (회색만 6종: `#475569`, `#607184`, `#64748b`, `#66768a`...) |
| 2 | **간격 체계 없음** | 4, 6, 8, 9, 10, 11, 12, 14, 18px 등 임의 값 | 정렬감 부족, 시각적 노이즈 |
| 3 | **폰트 사이즈 비일관** | 동일 역할(필드 라벨)이 15px/16px, 소형 뱃지 11px/12px | 미묘한 불일관이 "정돈되지 않음" 인상 |
| 4 | **모달/오버레이 불통일** | position fixed vs absolute, opacity 0.28 vs 0.35 | 화면별 다른 느낌 |
| 5 | **그림자 불통일** | 모달 그림자 4종 (20px/18px/14px/12px) | 깊이감 불일관 |
| 6 | **border-radius 난립** | 3, 6, 7, 8, 10, 12, 50%, 999px | 소형 요소의 모서리가 제각각 |
| 7 | **Agent 배경 `#bbbbbb`** | 콘텐츠 영역이 너무 어두움 | 텍스트 대비 떨어지고 시각적으로 무거움 |
| 8 | **브릿지 연두색 배경** | `#f1f8ea`는 우리은행 CI와 무관 | 브랜드 톤에서 이탈 |
| 9 | **P7/P10 배경** | 여신 업무와 무관한 개인 고객 화면 | 스토리텔링 단절 |

## 2. 시연 캔버스 사양

### 2.1 화면 크기

| 항목 | 권장값 | 근거 |
|------|--------|------|
| **캔버스 크기** | **1920 × 1080 px** (Full HD) | 현대 프로젝터/모니터 기본 해상도. 현행 1280×720은 HD급으로 텍스트 선명도 부족 |
| **대체 캔버스** | 1280 × 720 px (유지 옵션) | 기존 코드 호환, 소형 프로젝터 대응 |
| **스케일 전략** | CSS `transform: scale()` 래핑 | 1920 기준 제작 → 720p 환경에서 자동 축소 |
| **녹화 대응** | 16:9 비율 고정, 좌우 safe-margin 40px | 녹화 영상 인코딩 시 가장자리 잘림 방지 |

### 2.3 가독성 기준 (시연 환경)

| 시연 환경 | 시청 거리 | 최소 폰트 | 권장 폰트 |
|----------|----------|----------|----------|
| 65인치 TV (회의실) | 2~4m | 16px | 18px+ |
| 빔 프로젝터 (100인치) | 3~10m+ | 18px | 22px+ |
| 15인치 노트북 (발표자) | 0.5m | 14px | 16px+ |
| 녹화 영상 (1080p) | - | 16px | 18px+ |

**결론**: 모든 본문 텍스트 최소 18px (프로젝터 10m+ 시청 기준), 핵심 데이터 20px 이상, 제목 28~44px, 라벨/캡션 최소 14px. 어떤 텍스트도 14px 미만 불가.

### 3.4 화면 유형별 배경색 방향

현행의 문제: 단말(`#eef3f8`), Agent(`#edf0f3`+`#bbbbbb`), 브릿지(`#f1f8ea`) — 미묘하게 다른 배경이 체계 없이 적용.

**개선 방향**: 배경색 차이를 **의미 있는 시각 구분**으로 재정의

| 화면 유형 | 배경 전략 | 배경색 | 헤더 악센트 | 설계 의도 |
|----------|----------|--------|-----------|----------|
| **단말** | 실제 WINI 통합단말 풀 크롬 | `#FFFFFF`(메인), `#EFF2F5`(콘텐츠) | 네이비 타이틀바(`#1B2A4A`) + 메뉴바 + 탭바 + 사이드바 + 상태바 | 실제 은행 통합단말과 동일한 인상 |
| **Agent** | 다크 반투명 시각화 | `#0C1A2E`(다크 네이비) + 도트 그리드 | `rgba(255,255,255,0.04)` 반투명 헤더, 글로우 효과 뱃지 | "AI 백그라운드 프로세스" — 실제 화면이 아님을 명확히 |
| **브릿지** | 검토/확인의 주의 환기 | `#F5F9FF` | `#FFFFFF` + Woori Deep Blue 상단 바(4px) | 사람이 확인하는 공간, 브랜드 강조 |
| **브릿지 미니** | 단말 위 오버레이 | 반투명 블러 `rgba(15,23,42,0.30)` | - | 기존 업무 흐름 위의 알림 레이어 |

## 4. 타이포그래피

### 4.1 서체 전략: 우리다움체 중심 설계

#### 우리다움체(Wooridaum) 개요

| 항목 | 내용 |
|------|------|
| 제작 | 우리금융그룹 공식 전용 서체 (2022년 공개) |
| 특징 | 고딕 기반, 둥근 형태로 친근하고 부드러운 느낌, 디지털 환경 최적화 |
| 굵기 | Light(L) / Regular(R) / Bold(B) 3종 |
| 파일 위치 | `renewal/assets/font/WooridaumL.ttf`, `WooridaumR.ttf`, `WooridaumB.ttf` |
| 라이선스 | 무료 배포, 상업적 사용 가능 (CI/BI 제작 제외) |

#### 서체 적용 우선순위

| 우선 | 서체 | 역할 | 적용 범위 |
|------|------|------|----------|
| **1순위** | **우리다움체 (Wooridaum)** | **주 서체** | 제목, 헤더, 뱃지, 버튼, 라벨, 일반 본문, 카드 텍스트 |
| 2순위 | Pretendard | 보조 서체 (데이터 특화) | 테이블 숫자, 금액, 날짜, ID, 모노스페이스 필요 구간 |
| 3순위 | "Malgun Gothic", sans-serif | 시스템 폴백 | 우리다움체 미로드 시 대체 |

#### 우리다움체를 주 서체로 사용하는 근거

1. **브랜드 일체감**: 우리은행이 직접 배포한 공식 서체를 시연에 사용함으로써, 고객사(우리은행) 담당자가 "우리 서체"를 즉시 인지 → 제안 신뢰도 상승
2. **디지털 최적화**: 화면 표시를 전제로 설계된 서체이므로 렌더링 품질이 인쇄용 서체보다 우수
3. **둥근 고딕 특성**: 딱딱한 업무 시스템 톤을 부드럽게 완화 → 시연의 "AI가 도와주는" 느낌과 부합
4. **차별화**: 일반 시스템 폰트(맑은 고딕)가 아닌 전용 서체 사용으로 시연의 완성도 격상

#### 우리다움체 적용 시 유의 사항

| 유의 사항 | 대응 방안 |
|----------|----------|
| **굵기 3단계 한정** (L/R/B) | CSS `font-weight` 매핑: L→300, R→400, B→700. 중간 굵기(500, 600)는 R과 B 사이에서 브라우저가 보간하므로, 디자인상 L/R/B 3단계만 명시적 사용 권장 |
| **소형 텍스트 가독성** | 16px 이하에서는 둥근 형태가 뭉개질 수 있음 → 14px 텍스트(타임스탬프, Evidence ID)에는 Pretendard 또는 시스템 폰트 사용. 모든 텍스트 최소 14px |
| **숫자/금액 정렬** | 우리다움체의 숫자는 프로포셔널(글자 폭 가변) → 테이블의 금액 열 정렬이 어긋날 수 있음 → 금액/숫자 밀집 테이블 열에는 `font-variant-numeric: tabular-nums` 또는 Pretendard 적용 |
| **영문 혼용** | 영문 텍스트(Agent, HITL, OCR, BPR 등)에서 한영 밸런스 확인 필요 → 우리다움체의 영문이 자연스럽지 않을 경우 영문만 Pretendard 지정 |
| **웹폰트 변환** | TTF → WOFF2 변환 필요 (파일 크기 약 50% 절감). 3종 합계 약 5.2MB(TTF) → WOFF2 약 2.5MB |

#### @font-face 선언 (구현 가이드)

```css
@font-face {
  font-family: "Wooridaum";
  src: url("../assets/font/WooridaumL.woff2") format("woff2"),
       url("../assets/font/WooridaumL.ttf") format("truetype");
  font-weight: 300;
  font-style: normal;
  font-display: swap;
}
@font-face {
  font-family: "Wooridaum";
  src: url("../assets/font/WooridaumR.woff2") format("woff2"),
       url("../assets/font/WooridaumR.ttf") format("truetype");
  font-weight: 400;
  font-style: normal;
  font-display: swap;
}
@font-face {
  font-family: "Wooridaum";
  src: url("../assets/font/WooridaumB.woff2") format("woff2"),
       url("../assets/font/WooridaumB.ttf") format("truetype");
  font-weight: 700;
  font-style: normal;
  font-display: swap;
}
```

#### font-family 스택

```css
:root {
  /* 주 서체: 우리다움체 우선 */
  --font-primary: "Wooridaum", "Pretendard", "Malgun Gothic", sans-serif;

  /* 데이터 서체: 숫자/금액 테이블 특화 */
  --font-data: "Pretendard", "Wooridaum", "Malgun Gothic", sans-serif;

  /* 소형 텍스트 서체: 11px 이하 메타 정보 */
  --font-micro: "Pretendard", "Malgun Gothic", sans-serif;
}

body {
  font-family: var(--font-primary);
}
```

### 4.3 타입 스케일

현행의 문제: 9px, 11px, 12px, 13px, 14px, 15px, 16px, 20px, 22px, 24px, 28px — **12단계 이상** 무질서.

**개선**: 우리다움체 특성에 맞춘 **6단계** 스케일 (우리다움체는 둥근 형태라 일반 고딕보다 약간 크게 보임 → 0.5~1px 작게 설정해도 동일한 시인성)

| 토큰 | 크기 | 행간 | 권장 굵기 | 서체 | 용도 |
|------|------|------|----------|------|------|
| `--text-3xl` | 44px | 52px | B(700) | 우리다움 | 화면 대제목 (프로젝터 강조용) |
| `--text-2xl` | 36px | 44px | B(700) | 우리다움 | 핵심 수치, 강조 제목 |
| `--text-xl` | 28px | 36px | B(700) | 우리다움 | **헤더 타이틀** — 우리다움 Bold의 핵심 무대 |
| `--text-lg` | 22px | 30px | B(700)/R(400) | 우리다움 | 섹션 제목, 카드 헤더 |
| `--text-md` | 18px | 28px | R(400)/B(700) | 우리다움 | **기본 본문**, 입력값, 테이블 셀 |
| `--text-sm` | 16px | 24px | R(400) | 우리다움 | 보조 정보, 뱃지 텍스트, 소형 라벨 |
| `--text-xs` | 14px | 20px | - | **Pretendard** | 타임스탬프, Evidence ID — 프로젝터 최소 가독 크기 |

**시연 환경 최소 크기 규칙** (프로젝터 10m+ 시청 기준 전면 업스케일):
- 시연 화면 본문: `--text-md` (18px) 이상
- 핵심 데이터 값: `--text-lg` (22px) 이상
- 제목/헤더: `--text-xl` (28px) ~ `--text-3xl` (44px)
- 우리다움체 최소 사용 크기: **16px** (이하는 Pretendard 전환)
- 어떤 텍스트도 **14px 미만 불가** (기존 11px 하한 → 14px으로 상향)

### 4.4 굵기 체계

우리다움체 3종(L/R/B)에 맞춘 3단계 기본 + CSS 보간 1단계:

| 토큰 | CSS 값 | 우리다움체 매핑 | 용도 |
|------|--------|-------------|------|
| `--font-light` | 300 | **WooridaumL** | 배경 설명, 캡션, 부연 텍스트 |
| `--font-regular` | 400 | **WooridaumR** | 본문, 라벨, 입력값, 테이블 셀 |
| `--font-bold` | 700 | **WooridaumB** | 타이틀, 강조, CTA 버튼, 핵심 수치 |

> **참고**: `font-weight: 500` 또는 `600`을 지정하면 브라우저가 Regular(400)과 Bold(700) 사이를 보간하는데, TTF/WOFF2에서는 faux bold가 발생할 수 있다. 따라서 디자인 시 명시적으로 **300, 400, 700만 사용**하는 것을 원칙으로 한다. 시각적으로 중간 강조가 필요할 경우 Regular + 색상 강조(Woori Blue 등)로 대체한다.

## 12. 시연 안정성 설계 (오동작 방지)

> 시연은 **단 한 번의 실수도 허용되지 않는** 라이브 환경이다.  
> 의도치 않은 사용자 조작, 브라우저 동작, OS 이벤트로 인한 오동작을 원천 차단하는 방어적 설계가 필수이다.

### 12.1 위협 분류 및 대응 전략

| # | 위협 카테고리 | 구체적 시나리오 | 위험도 | 대응 전략 |
|---|-------------|---------------|--------|----------|
| T-01 | **의도치 않은 클릭** | 발표자가 화면 설명 중 마우스로 UI 요소를 클릭 | 최상 | 자동 재생 중 클릭 이벤트 무시 레이어 |

### 12.2 입력 차단 레이어 (Shield Layer)

자동 재생 모드에서 발표자의 의도치 않은 입력을 차단하는 투명 오버레이.

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│           [Shield Layer - z-index: 9999]              │
│           투명, pointer-events: auto                  │
│           모든 클릭/터치를 흡수                         │
│                                                      │
│  ┌──────────────────────────────────────────────┐    │
│  │          실제 시연 화면 콘텐츠                  │    │
│  │          pointer-events: none (쉴드 활성 시)   │    │
│  └──────────────────────────────────────────────┘    │
│                                                      │
│  [시연 컨트롤 영역 - 쉴드 제외]                        │
│  → / ← / Space / Esc 키만 통과                       │
└──────────────────────────────────────────────────────┘
```

**동작 원칙**:
- `IS_AUTO_PAGING = true` 시: Shield Layer 활성 → 화면 내 모든 클릭/터치 이벤트 흡수
- `IS_AUTO_PAGING = false` 시: Shield Layer 비활성 → 수동 조작 가능 (P8 HITL 값 수정 등)
- 시연 컨트롤 키(`→`, `←`, `Space`, `Esc`)는 Shield와 무관하게 항상 동작
- 데모 시퀀스가 내부적으로 호출하는 `click()`은 프로그래밍 방식이므로 Shield 영향 없음

### 12.7 데모 시퀀스 안정성

#### 시퀀스 실행 잠금 (Mutex)

```javascript
let _sequenceLock = false;

async function runDemoSequence() {
  if (_sequenceLock) return;    // 이중 실행 방지
  _sequenceLock = true;
  try {
    // ... 실제 데모 시퀀스 ...
  } catch (err) {
    console.error('[Demo] Sequence error:', err);
    recoverFromError();         // 자동 복구
  } finally {
    _sequenceLock = false;
  }
}
```

#### 애니메이션 완료 보장

```javascript
// 페이지 전환 전 진행 중인 애니메이션 강제 완료
function forceCompleteAnimations() {
  document.getAnimations().forEach(anim => {
    anim.finish();
  });
}

// 전환 시
async function safePageTransition(nextUrl, delayMs) {
  forceCompleteAnimations();
  await sleep(50);              // 강제 완료 후 안정화 대기
  WBPageTransition.fadeOutAndNavigate(nextUrl, delayMs);
}
```

#### 에러 자동 복구

```javascript
function recoverFromError() {
  // 1) 현재 페이지의 최종 상태로 강제 전환
  setAllBadgesToFinalState();

  // 2) 3초 대기 후 다음 페이지로 자동 이동

    WBAutoPaging.moveToNext(0);
  }, 3000);
}

// 전역 에러 핸들러
window.addEventListener('error', (e) => {
  if (isDemoRunning) {
    e.preventDefault();
    recoverFromError();
  }
});

  if (isDemoRunning) {
    e.preventDefault();
    recoverFromError();
  }
});
```
