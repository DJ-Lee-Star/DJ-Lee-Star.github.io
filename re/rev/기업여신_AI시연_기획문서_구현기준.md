# 기업여신 AI 시연 기획문서

## 1. 문서 목적
본 문서는 `new/html` 하위에 구현된 시연 화면을 기준으로, 제안 설명회용 기업여신 AI 시연의 기획 내용을 재정리한 문서다.  
`new/docs` 하위 문서는 참고 자료로만 활용하며, 본 문서는 실제 구현된 HTML/JS의 화면 흐름과 표현을 우선 기준으로 작성한다.

본 문서의 목적은 다음과 같다.

- 현재 시연이 전달하는 핵심 메시지와 범위를 명확히 정의한다.
- 화면별 역할과 시연 스토리를 업무 관점에서 정리한다.
- 향후 수정, 보완, 재구현 시 유지해야 할 핵심 사용자 경험 기준을 정리한다.

---

## 2. 시연 개요

### 2.1 시연명
기업여신 신청 서류입력 및 검수 자동화 시연

### 2.2 시연 한 줄 설명
영업점 담당자가 기업여신 신청정보와 제출서류를 등록하면, AI Agent가 서류를 자동 분석, 검증, 추출하고 사람이 필요한 항목만 확인 및 수정한 뒤 최종 심사요청까지 이어지는 흐름을 보여주는 시연이다.

### 2.3 시연의 핵심 메시지
- 신청 접수와 서류 등록은 영업점 단말에서 수행된다.
- AI Agent는 문서 전처리, 진위검증, OCR 및 필드 추출, 대사 및 검증을 백그라운드에서 수행한다.
- AI는 저신뢰 또는 검토 필요 항목을 브릿지 화면으로 넘기고, 최종 판단은 사람이 수행한다.
- 최종 확정값은 다시 단말 화면의 심사요청 단계로 연결된다.

즉, 이 시연의 중심 메시지는 `완전 자동화`가 아니라 `자동 처리 + 사람 확정(HITL)`이다.

## 3. 시연 배경과 업무 맥락

### 3.1 RFP 관점의 배경
참고 문서 기준으로 이번 시연은 크게 두 축과 연결된다.

- 여신 특화 Agent
  - 기업여신 심사 과정에서 AI가 업무를 보조하는 실행형 시나리오
- 여신 서류 입력 및 검수 자동화
  - 서류 등록
  - 서류 식별 및 처리 분기
  - 진위 확인
  - 주요 값 추출
  - 추출값 검토 후 단말 입력 지원

### 3.2 실제 구현본의 초점
실제 구현 화면은 RFP 전체 범위를 모두 구현한 것이 아니라, 기업여신 서류 자동검토 파이프라인의 핵심 흐름을 중심으로 구성되어 있다.

1. 신청 기본정보 입력
2. 제출서류 등록
3. AI 처리 계획 수립
4. 전처리
5. 진위검증
6. OCR 및 필드 추출
7. 오류 알림
8. 사람 수정(HITL)
9. 대사 및 검증
10. 완료 알림
11. 최종 결과 확인
12. 심사정보 자동입력 및 심사 요청

## 4. 시연 목표

### 4.1 업무 목표
- 기업여신 신청 건의 서류 검토 업무를 자동화 가능한 흐름으로 제시한다.
- 담당자의 반복 입력과 대조 부담을 줄이는 업무 보조 경험을 보여준다.
- 오류 검출, 저신뢰 검토, 최종 심사 반영까지의 연결성을 강조한다.

### 4.2 제안 설명회 목표
- AI가 어디까지 자동으로 처리하고, 어디서 사람이 개입하는지 한 번에 이해시키는 것
- 현업 단말과 유사한 톤을 유지해 실제 적용 가능성을 높이는 것
- 단순 OCR 시연이 아니라, 검증과 확정까지 이어지는 E2E 흐름을 보여주는 것

### 4.3 향후 재구현을 위한 목표
- 현재 구현은 시연용 목업에 가까우므로, 이후 재구현 시 기능 확장과 구조 정비가 가능하도록 기준을 남긴다.
- 화면별 역할, 데이터 연결, 상태 흐름을 재사용 가능한 수준으로 정리한다.

## 7. 화면 유형 정의

### 7.1 단말
- 실제 은행 통합단말을 모티브로 한 업무 입력형 화면
- 사용자의 명시적 액션이 존재
- 폼, 조회, 테이블, 버튼 중심

### 7.2 Agent
- AI 내부 처리 상태를 보여주는 운영형 화면
- 사용자가 직접 조작하지 않으며, AI가 무엇을 처리 중인지 보여주는 목적
- 상단 단계 플로우와 하단 상태 영역으로 구성

### 7.3 브릿지
- AI 결과를 사람이 검토, 수정, 확정하는 중간 화면
- 좌측 문서 목록과 우측 상세 결과 구조
- HITL의 핵심 채널

### 7.4 브릿지 미니
- 우측 하단 메시지형 알림 UI
- 업무 중간 개입 요청 또는 완료 통지에 사용

## 9. 화면별 기획 상세

### P1. 신청정보 입력
#### 목적
- 기업여신 신청 건의 기본정보를 등록한다.

#### 구현 기준 주요 항목
- 차주명(법인)
- 신청번호
- 영업점 코드
- 담당자
- 신청일자
- 대출상품
- 신청금액
- 만기일
- 상환방식
- 취급구분
- 자금용도
- 담보유형
- 신청채널

#### 시연 포인트
- 영업점 코드와 담당자는 조회 팝업에서 선택한다.
- 일부 항목은 자동 타이핑, 일부는 선택형 드롭다운으로 시연된다.
- 단순 입력이 아니라 `업무 건 생성`의 시작점 역할을 한다.

### P2. 첨부문서 등록
#### 목적
- 신청 건에 필요한 제출서류를 등록한다.

#### 구현 기준 문서 목록
- 사업자등록증명원
- 법인등기부등본
- 표준재무제표증명
- 부가가치세 과세표준증명원
- 법인인감증명서
- 주주명부(미등록 상태)

#### 시연 포인트
- 파일선택 모달을 열고 문서를 등록한다.
- 등록현황이 갱신된다.
- BPR전송 완료 팝업이 표시된다.
- 최종적으로 `서류입력 및 검수(AI)` 단계로 이어지는 출발점이다.

### P3. 실행 계획 수립
#### 목적
- 업로드된 문서를 일괄 처리하지 않고, 문서 종류에 따라 어떤 작업을 수행할지 결정한다.

#### 시연 포인트
- 문서 전처리 대상
- AI-OCR 수행 대상
- 진위검증 대상
- 대사 및 검증 대상

#### 전달 메시지
- AI는 문서별 속성에 따라 작업을 분기한다.
- 전체 처리 전에 실행 계획을 세운다.

### P4. 문서 전처리
#### 목적
- OCR 전 단계에서 이미지 품질을 보정한다.

#### 구현 기준 작업
- 기울기 보정
- 노이즈 제거
- 명암 및 대비 보정
- 페이지 정렬

#### 전달 메시지
- OCR 정확도는 원본 품질의 영향을 받기 때문에 사전 정리가 필요하다.
- 문서별로 필요한 전처리 작업은 다를 수 있다.

### P5. 진위검증
#### 목적
- 제출서류가 실제 발급기관 기준으로 유효한지 확인한다.

#### 구현 기준 대상 문서
- 사업자등록증명원
- 법인등기부등본
- 법인인감증명서

#### 구현 기준 세부 단계
- 기관 API 호출
- 응답전문 파싱
- 진위 확인

#### 전달 메시지
- AI는 문서를 읽는 것만이 아니라, 외부 기준으로 진짜인지도 확인한다.
- 자동화된 진위확인은 심사 신뢰도를 높이는 핵심이다.

### P6. OCR 및 필드 추출
#### 목적
- 문서별 핵심 필드를 자동 추출하고, 신뢰도를 평가한다.

#### 구현 기준 특징
- 문서별 처리 단계
  - 텍스트 인식
  - 필드 매핑
  - 신뢰도 평가
- `법인등기부등본.pdf`에서 `검토필요` 상태가 발생한다.
- 메시지 전송 툴을 호출해 사용자 확인 요청 이벤트를 생성한다.

#### 전달 메시지
- AI는 대부분 항목을 자동으로 처리하되, 저신뢰 항목은 사람에게 넘긴다.
- 자동화의 목표는 무조건 통과가 아니라, 검토가 필요한 항목을 빠르게 좁히는 것이다.

### P7. 오류 알림
#### 목적
- 사용자가 지금 개입해야 할 이슈가 생겼음을 빠르게 알린다.

#### 구현 기준 메시지
- 법인등기부등본 신뢰도평가 검토필요 1건 발생

#### 전달 메시지
- Agent의 내부 이벤트가 실제 업무 알림으로 연결된다.
- 사용자는 모든 결과를 일일이 열어보지 않고도 개입 시점을 알 수 있다.

### P8. 값 수정(HITL)
#### 목적
- 사용자가 원문 근거를 확인하고, AI 추출값을 검토 및 수정한다.

#### 핵심 시연 시나리오
- 대상 문서: `법인등기부등본.pdf`
- 대상 항목: `자본금`
- 변경 전 값: `1,000,000,000원`
- 변경 후 값: `1,600,000,000원`
- 변경사유: `OCR 오인식(스캔 품질)`

#### 전달 메시지
- 최종 판단 책임은 사람에게 있다.
- AI는 원문 근거와 추출값을 연결해 사람의 판단을 돕는다.

### P9. 대사 및 검증
#### 목적
- 입력정보와 문서정보, 문서 간 값들을 최종적으로 대조한다.

#### 구현 기준 검증 항목
- 입력값 대사
- 문서간 대사
- 필수값 누락 검증
- 타입 검증
- 유효값 검증
- KYC 교차검증
- 최종 판정

#### 전달 메시지
- OCR 추출만으로 끝나지 않고, 업무 입력값과의 정합성 검증까지 이어진다.
- 자동 수정이 끝난 뒤에도 최종 검증 단계가 필요하다.

### P10. 완료 알림
#### 목적
- 사용자가 최종 결과를 검토할 시점임을 알린다.

#### 전달 메시지
- AI 처리가 완료되었고, 이제 사용자 확정 단계로 넘어간다.

### P11. 결과 확인(HITL)
#### 목적
- 사용자가 단계별 처리 결과를 최종 검토하고 확정 직전 상태를 확인한다.

#### 구현 기준 특징
- 상단 단계 버튼을 눌러 각 처리 단계 결과 확인 가능
- 문서별 상태와 OCR 결과를 다시 탐색 가능
- 최종 액션 버튼: `WINI 전송`

#### 전달 메시지
- 단순 수정 1건만 처리하는 것이 아니라, 전체 과정을 사람이 이해하고 승인할 수 있어야 한다.

### P12. 심사정보 자동입력 및 심사 요청
#### 목적
- 확정된 정보를 단말에 반영하고 심사 요청을 수행한다.

#### 구현 기준 노출 항목
- 신청 기본정보
- 사업자등록번호
- 법인등록번호
- 대표자
- 본점주소
- 자본금

## 12. 현재 구현의 강점
- 12개 화면이 하나의 선형 시연 흐름으로 자연스럽게 이어진다.
- 단말, Agent, 브릿지의 역할 구분이 명확하다.
- `AI 자동 처리 -> 오류 발견 -> 사람 수정 -> 최종 반영` 스토리가 분명하다.
- 설명회 환경에서 이해하기 쉬운 가시적 애니메이션과 상태 표현을 사용한다.
- `P8`, `P11` 브릿지 화면은 향후 재구현 시 핵심 UX 자산으로 활용 가능하다.

---

## 13. 현재 구현의 한계
- 모든 데이터가 페이지별 하드코딩이라 실제 연계 구조가 없다.
- 동일 데이터가 여러 화면에 중복 정의되어 유지보수성이 낮다.
- 일부 문구와 제목은 실제 의미와 어긋나는 표현이 있다.
- 오류 발생 후 재시도, 반송, 보완 요청 등 업무 예외 흐름이 부족하다.
- 문서 처리 결과의 이력 관리와 사유 관리가 시연 수준에 머무른다.

## 14. 재구현 시 반드시 유지할 기획 요소
- 신청정보 입력부터 심사요청까지 이어지는 E2E 내러티브
- Agent 처리 흐름을 상단 단계 플로우로 시각화하는 방식
- 브릿지 미니를 통한 오류 및 완료 이벤트 전달
- 브릿지에서 원문 근거와 추출값을 함께 보는 HITL 구조
- 저신뢰 항목을 사람이 수정하고 최종 단말 반영으로 이어지는 스토리

---

## 15. 재구현 시 보강이 필요한 기획 요소
- 단계별 성공, 실패, 재시도, 반송 시나리오 추가
- 문서 종류와 처리 규칙의 체계화
- 사용자 역할별 책임 구분 명확화
- 최종 확정, 예외 유지, 반송 사유 입력 체계 보강
- 실제 연계 가능한 데이터 모델과 이벤트 모델 정의
