# 기업여신 AI 시연 구현 불일치 항목 목록

## 1. 문서 목적
본 문서는 현재 구현된 `new/html` 시연 화면을 기준으로, 다음 세 가지 관점을 함께 비교해 정리한 문서다.

- 현재 구현 상태
- RFP 요구 취지 및 기대 수준
- 재구현 시 반드시 보강해야 할 조치

이 문서의 목적은 단순히 구현의 부족한 점을 지적하는 것이 아니라, 향후 재구현 범위를 명확히 정의하고 우선순위를 합의 가능한 형태로 정리하는 데 있다.

---

## 2. 분류 기준

| 구분 | 설명 |
|---|---|
| 용어 | 화면 제목, 버튼명, 상태명, 문서명 등의 명칭 불일치 |
| 화면 | 화면 역할과 실제 표현의 불일치 |
| 데이터 | 도메인 데이터, 문서 데이터, 추출 데이터의 누락 또는 중복 |
| 상태 | 업무 상태와 시연 상태의 혼재 |
| 시나리오 | RFP 요구 프로세스 대비 누락 또는 축소된 흐름 |
| 연계 | 단말, BPR, WINI, RPA, 이메일/팩스 등 연계 표현 수준 |
| 통제 | HITL, 이력, 예외, 반송, 보안 등 내부통제 관련 항목 |

우선순위는 다음과 같이 정의한다.

- `상`: 재구현 시 1차 반영 필수
- `중`: 핵심 시연을 안정화한 후 2차 반영
- `하`: 확장 또는 고도화 단계에서 반영

---

## 3. 핵심 요약

현재 구현은 `서류 등록 -> AI 처리 -> HITL 수정 -> 심사요청`이라는 핵심 시나리오는 분명하게 보여준다.  
반면 RFP 관점에서 보면 다음 영역이 구조적으로 부족하다.

- RFP가 요구하는 Agent의 `제언`, `QA`, `업무 맥락 설명` 기능이 거의 표현되지 않음
- 서류 자동검토 흐름은 있으나 `필요 서류 제언`, `누락 서류 안내`, `공유주주 누락 점검`, `특약/유의사항 안내`가 없음
- 단말 연계와 WINI 입력 지원은 최종 액션 수준으로만 존재하고, 실제 데이터 전송 구조는 설계적으로 연결되어 있지 않음
- 예외 처리와 내부통제 구조가 `검토필요 1건 수정` 수준에 머물러 있음

즉, 현재 구현은 `자동검토 시연`으로서는 유효하지만, `RFP 충실 반영형 재구현`을 위해서는 시나리오, 데이터, 상태, 연계 설계를 모두 확장해야 한다.

## 4. 불일치 항목 목록

### GAP-01
- 구분: 용어
- 위치: `P7`, `P10`
- 현재 구현:
  - `p7.html` title은 `검토 요청 알림`인데 화면 헤더는 `검토 완료 알림`
  - `p10.html` title은 `검토 완료 알림`인데 화면 헤더는 `검토 요청 알림`
- RFP 및 기획 기대:
  - 오류 알림과 완료 알림은 업무 이벤트 성격이 명확히 구분되어야 함
- 영향:
  - 발표 자료, 화면 설명, 후속 설계 문서에서 혼선 발생
- 재구현 조치:
  - 알림 이벤트 유형을 `오류`, `보완요청`, `완료`, `승인대기` 등으로 표준화
  - 파일명, title, 헤더, 문서 표현을 동일 기준으로 정렬
- 우선순위: `상`

### GAP-02

- 위치: `P6`, `P9`
- 현재 구현:
  - `메세지 전송 Tool` 표기를 사용
- RFP 및 기획 기대:
  - 대외/대내 설명 문서에서는 표기 일관성이 필요
- 영향:
  - 시연 완성도 저하, 문서 신뢰감 저하
- 재구현 조치:
  - `메시지 전송 Tool` 또는 기능 중심 명칭으로 표준화
- 우선순위: `중`

### GAP-03
- 구분: 화면
- 위치: `P7`, `P10`
- 현재 구현:
  - 브릿지 미니 알림 배경으로 일반 PB성 화면이 노출되어 여신 검토 업무 맥락이 약함
- RFP 및 기획 기대:
  - 공통서비스 화면 내 여신 업무 흐름이 중심이어야 함
- 영향:
  - 알림은 맞지만 “기업여신 자동검토” 시연이라는 맥락이 약해짐
- 재구현 조치:
  - 알림 배경을 여신 신청/검토 화면 또는 공통서비스 대시보드로 교체
  - 알림 payload도 신청번호, 문서명, 이슈유형, 요청 액션 중심으로 재구성
- 우선순위: `상`

### GAP-04
- 구분: 화면
- 위치: `P8`, `P11`
- 현재 구현:
  - 일부 단계 선택 시 실제 상세 뷰 대신 placeholder 문구가 노출됨
- RFP 및 기획 기대:
  - 검토/입력 전 단계에서 서류 이미지와 입력값 하이라이팅을 통한 검토 지원이 필요
- 영향:

- 재구현 조치:
  - `대사/검증`, `실행 계획`, `전처리`, `진위검증`도 실제 검토 가능한 상세 뷰로 구현
  - 단계별 근거, 결과, 담당 액션을 명확히 분리
- 우선순위: `상`

### GAP-05

- 위치: 전 화면
- 현재 구현:
  - 신청정보, 문서정보, 추출값이 각 페이지에 하드코딩 중복 정의됨
- RFP 및 기획 기대:

  - 한 화면의 수정이 다른 화면과 설계적으로 연결되지 않음
- 재구현 조치:
  - 단일 신청 건 데이터 모델 정의
  - 문서 목록, 추출값, 검증결과, 수정이력, 심사반영값을 하나의 시나리오 데이터로 통합
- 우선순위: `상`

- 영향:

- 재구현 조치:
  - 주주명부 문서를 실제 처리 대상에 포함
  - `공유주주 누락 점검`, `실소유자/KYC 교차검증` 별도 시나리오 추가
- 우선순위: `상`

  - 예: `법인등기부등본 자본금 불일치 -> 조치 권고안`, `누락 서류 -> 제출 안내`

  - 문서간 불일치 또는 수기입력 충돌

  - 검토 결과와 함께 `유의사항` 또는 `상향검토 권고` 표시
- 우선순위: `중`

- 재구현 조치:
  - 연계 대상별 인터페이스 정의서 작성
  - 입력 전 검토값, 확정값, 전송 payload, 전송 결과 상태를 분리

- 우선순위: `상`

### GAP-16
- 구분: 화면

  - 단말 입력 필드와 문서 근거의 연결 표시 추가
  - 변경된 필드 강조, 검토완료 여부, 전송 전 확인 UI 추가
- 우선순위: `상`

### GAP-17
- 구분: 시나리오
- 위치: 전체
- 현재 구현:
  - 성과지표, 모니터링, 동시접속/성능 고려는 시연에 포함되지 않음
- RFP 및 기획 기대:
  - 성과수치 Factor 수집 및 중앙화 모니터링 필요
- 영향:
  - 제안 설명 시 운영 관점 확장성이 약해 보일 수 있음
- 재구현 조치:
  - 운영 대시보드까지는 아니더라도, 처리건수/검토건수/보완건수 같은 KPI 패널을 옵션 화면으로 설계
- 우선순위: `하`

## 5. 재구현 1차 반영 우선순위

다음 항목은 RFP 충실 반영형 재구현에서 1차 범위로 반드시 포함하는 것이 적절하다.

1. 알림 및 화면 용어 표준화
2. 단일 데이터 모델 도입
3. 필요 서류 판단 및 누락 안내
4. 주주명부 및 KYC 교차검증 시나리오 추가
5. HITL 확정, 예외유지, 보완요청 상태 모델 도입
6. WINI/BPR 전송 payload 및 상태 설계
7. P8, P11의 상세뷰 실구현
8. 단말 입력값과 문서 근거의 하이라이팅 검토 UX 강화

## 6. 문서 활용 방식
이 문서는 향후 아래 순서로 활용하는 것이 적절하다.

## 7. 결론
현재 구현은 기업여신 서류 자동검토의 핵심 흐름을 이해시키는 데는 충분하지만, RFP를 충실히 반영한 재구현 관점에서는 `필요 서류 판단`, `제언`, `공유주주/KYC`, `연계 payload`, `HITL 통제`, `보완 루프`가 추가로 설계되어야 한다.

따라서 재구현은 단순 화면 개선이 아니라, 본 문서의 불일치 항목을 기준으로 시나리오, 데이터, 상태, 연계 구조를 동시에 재정의하는 작업으로 접근하는 것이 적절하다.
