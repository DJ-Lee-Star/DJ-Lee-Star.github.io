# 기업여신 AI 시연 G1-G8 화면상세명세서

## 1. 문서 목적
본 문서는 우리은행 기업여신 AI 시연의 목표 화면 G1~G8에 대한 상세 역할을 정의한다.
이번 버전은 `1920 x 1080` 해상도와 `단말 -> Agent -> 브릿지 -> 단말` 업무 흐름을 기준으로 작성한다.

---

## 2. 공통 전제
- 해상도: `1920 x 1080` 고정
- 시연모드: 수동 진행 기준, 자동 진행은 보조 옵션
- 단말은 AI 기능 직접 사용 불가
- 단말은 AI 분석 요청만 수행
- Agent는 요청 건을 처리하고 검토 필요 건만 브릿지로 전달
- 브릿지는 승인, 수정, 보완 판단을 수행
- 최종 결과는 단말에 반영되어 후속 업무로 이어짐
- 대표 시연 시나리오는 `부동산 담보 기업대출`이다.
- 담보 관련 필수 검토 대상에는 `등기부`, `부동산 권리관계`, `담보가치 산정 근거`, `사정의견서 또는 추정가액조사표`가 포함된다.
- AI 제언은 단말을 제외한 여러 단계에 분산 배치하며, 최소 `수정 제안`, `보완 조치안`, `담보 유의사항 안내`를 포함한다.

---

## 3. 화면 상세

## G1. 신청정보 입력
### 목적
- 영업점 단말에서 기업여신 신청 기본정보를 입력한다.
- 제출서류 등록에 필요한 상품 기준 정보를 확인한다.

### 화면군
- 단말

### 주요 데이터
- 차주명
- 신청번호
- 영업점 코드
- 담당자
- 대출상품
- 신청유형
- 신청금액
- 담보유형
- 담보물 소재지
- 담보예정가액
- 자금목적
- 상환방식
- 신청채널

### 주요 출력
- 상품 기준 제출서류 안내
- 담보 유형별 등록 참고 정보
- 단말 상단 브랜드/단말번호 정보

### 주요 액션
- 등록 기준 보기
- 다음 단계 이동

### 시연 포인트
- 단말은 신청 정보 입력과 등록 준비까지만 담당한다.
- 이 화면에서 AI 분석을 실행하는 것이 아니라, 이후 서류 등록 및 요청 단계로 넘긴다.
- 제출서류 기준은 단말의 업무 기준 정보이며, 추가 검토나 보완 필요 판단은 Agent와 브릿지에서 수행한다.
- 화면 설명용 카피보다 실제 단말에 있을 법한 조회 결과와 입력 항목 중심으로 구성한다.

---

## G2. 첨부서류 등록 및 AI 분석 요청
### 목적
- 영업점 단말에서 서류 등록 현황을 확인한다.
- 필수/조건부 서류 제출 상태를 관리한다.
- Agent로 AI 분석 요청을 보낸다.

### 화면군
- 단말

### 주요 데이터
- 서류명
- 제출 여부
- 파일명
- 접수 시각
- 조건부 여부
- 누락 여부
- 담보 서류 여부

### 주요 출력
- 제출 완료/미제출 상태
- 보완 후보 요약
- 접수 타임라인
- 요청 대상 건 정보
- 담보 관련 서류 누락 경고

### 주요 액션
- 누락 사유 보기
- AI 분석 요청

### 시연 포인트
- 영업점 단말에서 직접 AI 화면으로 들어가는 것이 아니라, 분석 요청을 시스템에 전달하는 구조를 보여 준다.
- "AI 시작"보다 "AI 분석 요청" 표현을 사용한다.
- 담보 관련 서류가 부족하더라도 단말은 상태만 보여 주고, 후속 보완 필요 판단은 브릿지 검토에서 수행한다.

---

## G3. Agent 처리 현황
### 목적
- 요청된 건에 대해 Agent가 수행하는 자동 처리 현황을 보여 준다.
- OCR, 정합성 검증, 이상 징후 탐지, 제언 생성 상태를 통합 표현한다.
- 실제 업무 사용자가 조작하는 화면이 아니라, 백그라운드 자동 처리 상태를 시연용으로 관찰하는 모니터 뷰로 정의한다.

### 화면군
- Agent

### 주요 데이터
- 처리 대상 문서 목록
- 전처리 상태
- OCR 추출 결과
- 검증 결과
- 검토 요청 후보
- AI 제언
- 담보가치 산정 보조 결과
- 권리관계 이상 징후

### 주요 출력
- 처리 진척률
- 이상 징후 건수
- 검토 요청 건수
- 브릿지 전달 사유
- 담보 관련 조치 권고안
- 내부 처리 로그와 단계별 상태
- "실제 사용자 조작 없음" 안내 배너

### 주요 액션
- 시연 기준으로는 별도 사용자 액션 없음
- 상태 확인 중심

### 시연 포인트
- Agent는 별도 처리 채널이다.
- 사람은 여기서 직접 값을 수정하지 않는다.
- 실제 서비스에서는 화면이 없어도 되는 백그라운드 처리이며, 시연에서는 이해를 돕기 위해 모니터 형태로만 노출한다.
- 검토가 필요한 항목만 브릿지로 전달된다는 점을 강조한다.
- 자본금 불일치와 함께 `담보 권리관계/담보가치 확인 필요`를 AI 제언으로 제시해, RFP의 Action Plan 성격을 보여 준다.

---

## G4. 브릿지 검토 요청 알림
### 목적
- Agent가 브릿지 검토가 필요한 건을 전달했음을 알린다.
- 브릿지 사용자가 어떤 이슈를 확인해야 하는지 즉시 파악하게 한다.

### 화면군
- 브릿지 미니

### 주요 출력
- 신청번호
- 검토 요청 사유
- 대상 항목 수
- 바로가기 액션
- 대표 Action Plan 요약

### 주요 액션
- 브릿지 검토 화면 열기

### 시연 포인트
- 알림은 개별 전체 화면보다 오버레이형 UI가 적절하다.
- 전달 사유는 짧고 명확해야 한다.

---

## G5. 브릿지 검토 및 수정 반영
### 목적
- 브릿지 사용자가 근거 문서를 보고 AI 추출값을 검토한다.
- 수정, 승인, 보완 요청 여부를 결정한다.
- 단말 반영 가능 상태를 만든다.

### 화면군
- 브릿지

### 주요 데이터
- 검토 대상 필드
- 원 추출값
- 비교 기준값
- 문서 근거
- 제언 메시지
- 수정 입력값
- 담보 검토 의견

### 주요 출력
- 승인/수정/보완 상태
- 반영 대상 값
- 브릿지 판단 사유
- 담보 관련 보완/한도 조정 검토 의견

### 주요 액션
- 수정 반영 승인
- 보완 요청 결정
- 다음 검토로 이동

### 시연 포인트
- 사람의 판단이 필요한 이유를 보여 주는 핵심 화면이다.
- 브릿지는 단순 조회가 아니라 통제 지점이다.
- 담보평가나 권리관계 이슈가 있으면 단순 값 수정이 아니라 조치안 판단까지 이어진다는 점을 강조한다.

---

## G6. 대사/검증 및 주주명부/KYC/담보 확인
### 목적
- 주주명부, KYC, 대사 결과를 교차 검토한다.
- 보완 요청이 필요한지 최종 판단한다.
- 부동산 담보 관련 서류를 바탕으로 권리관계, 담보가치, 담보 유의사항을 검토한다.

### 화면군
- 브릿지 확장 또는 Agent 보조

### 주요 데이터
- 주주명부 제출 여부
- KYC 확인 상태
- 대주주 관련 검증 결과
- 교차 검증 결과
- 담보물 소재지
- 담보가치
- 선순위 권리 여부
- 감정/사정 의견 요약

### 주요 출력
- 일치/불일치 판정
- 보완 필요 여부
- 위험 요약
- 담보 검토 결과
- AI Action Plan

### 주요 액션
- 보완 요청 확정
- 검증 통과 처리
- 담보 추가 확인 요청
- 한도/조건 조정 검토

### 시연 포인트
- RFP 상 차별화 포인트다.
- 단순 OCR이 아니라 심사 보조형 검증이라는 메시지를 강화한다.
- 실제소유자 검증과 함께 부동산 담보 검토를 묶어 보여 주면, 기업여신 심사 보조 Agent의 실무성을 설명하기 좋다.

---

## G7. 단말 반영 결과 확인
### 목적
- 브릿지에서 승인/수정한 결과가 단말에 반영되었음을 보여 준다.
- 영업점이 직접 AI 결과를 건드린 것이 아니라 통제 절차를 거쳐 반영되었음을 보여 준다.

### 화면군
- 단말

### 주요 데이터
- 반영 필드 목록
- 변경 전 값
- 변경 후 값
- 반영 시각
- 반영 사유

### 주요 출력
- 반영 완료 메시지
- 수정된 필드 강조
- 후속 업무 가능 상태
- 담보 검토 결과 반영 내역

### 주요 액션
- 반영 상세 보기
- 심사 요청 화면 이동
- 보완 요청 화면 이동

### 시연 포인트
- 브릿지 결과가 단말에 다시 적용되는 구조가 보여야 한다.
- 이 화면이 있어야 전체 흐름이 업무 시스템답게 닫힌다.

---

## G8. 심사 요청 또는 보완 요청 분기
### 목적
- 최종 결과에 따라 심사 요청 또는 보완 요청을 실행한다.

### 화면군
- 단말

### 주요 데이터
- 최종 상태
- 심사 요청 대상 값
- 보완 요청 사유
- 누락 서류/보완 필드

### 주요 출력
- 심사 요청 완료 메시지
- 보완 요청 생성 메시지
- 후속 상태값
- AI 제언 기반 체크리스트

### 주요 액션
- 심사 요청
- 보완 요청 생성
- 이전 단계 복귀

### 시연 포인트
- 단순 완료가 아니라 업무 분기까지 닫혀야 한다.
- 시연 결론은 "AI가 알아서 끝낸다"가 아니라 "업무 흐름이 끊기지 않는다"여야 한다.

---

## 4. 권장 시연 흐름
### 정상 흐름
G1 -> G2 -> G3 -> G4 -> G5 -> G7 -> G8(심사 요청)

### 보완 흐름
G1 -> G2 -> G3 -> G4 -> G5 -> G6 -> G7 -> G8(보완 요청)

---

## 5. 결론
G1~G8은 화면 수 자체보다 역할 분리가 중요하다.
향후 구현과 문서 보강은 다음 원칙을 유지해야 한다.
- 단말은 입력/등록/요청/반영 확인
- Agent는 자동 처리
- 브릿지는 검토와 판단
- 최종 결과는 다시 단말 업무로 복귀
