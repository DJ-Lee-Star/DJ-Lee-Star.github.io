# 기업여신 AI 시연 RFP 반영 목표 화면 구성안

## 1. 문서 목적
본 문서는 우리은행 기업여신 AI 시연 화면을 재구성하기 위한 목표 화면 체계를 정의한다.
기존 구현 화면을 단순 보완하는 것이 아니라, RFP 취지와 실제 시연 운영 조건을 반영해 화면 수, 역할, 화면 간 관계를 다시 정리하는 것이 목적이다.

이번 개정에서는 다음 두 가지를 최우선 기준으로 반영한다.
- 시연 해상도 기준을 `1920 x 1080`으로 상향한다.
- 영업점 단말은 AI 기능을 직접 실행하지 않고, `단말 요청 -> Agent 처리 -> 브릿지 알림/검토 -> 단말 반영` 순서로 동작한다.
- 대표 시연 상품은 `부동산 담보 기업대출`로 고정하고, 담보서류 검증과 담보평가 지원 흐름을 포함한다.

---

## 2. 핵심 원칙

### 2.1 해상도 및 시연 운영 원칙
- 모든 주요 시연 화면은 `1920 x 1080` 고정 캔버스를 기준으로 설계한다.
- 발표 현장에서는 반응형보다 고정 레이아웃 안정성이 더 중요하다.
- 화면 하단 잘림, 스크롤, 브라우저 줌 이슈가 발생하지 않도록 화면 내 정보량을 통제한다.

### 2.2 업무 흐름 원칙
- 영업점 단말은 신청 정보 입력, 서류 등록, AI 분석 요청, 결과 반영 확인을 담당한다.
- 영업점 단말에서 AI Agent 기능을 직접 조작하지 않는다.
- AI Agent는 요청된 건을 비동기 처리하고, 검토가 필요한 이슈만 브릿지로 전달한다.
- 브릿지는 검토 알림 수신, 근거 확인, 수정/승인/보완 판단, 단말 반영 결정을 담당한다.
- 최종 승인 또는 보완 지시는 다시 단말 업무 흐름으로 연결된다.
- 대표 담보 시나리오는 `부동산 담보`이며, 등기부/대장/감정 또는 사정의견서/권리관계 문서를 함께 다룬다.

### 2.3 시연 메시지 원칙
- 메시지는 "단말에서 AI를 직접 쓰는 화면"이 아니라 "업무 시스템이 AI를 호출하고, 사람은 필요한 통제 지점에서 개입한다"로 정리한다.
- 핵심 시연 포인트는 자동화 그 자체보다 `선별`, `검토`, `반영 통제`, `업무 연결성`이다.
- AI 제언은 단말이 아니라 `Agent 처리`, `브릿지 검토`, `보완/심사 분기` 단계에 배치한다.

## 3. 목표 화면 체계 재정의
기존 G1~G8 체계를 유지하되, 역할과 연결 관계를 아래와 같이 명확히 정리한다.

| 화면 ID | 화면명 | 화면군 | 역할 |
|---|---|---|---|
| G1 | 신청정보 입력 | 단말 | 신청 기본정보 입력, 제출서류 등록 준비 |
| G2 | 첨부서류 등록 및 AI 분석 요청 | 단말 | 서류 등록, 제출 현황 관리, Agent 분석 요청 |
| G3 | 백그라운드 Agent 처리 모니터 | Agent | 실제 사용자 UI가 아닌 내부 처리 상태 관찰, 추출/검증/제언 생성 |
| G4 | 브릿지 검토 요청 알림 | 브릿지 미니 | Agent가 검토 필요 건을 브릿지로 전달 |
| G5 | 브릿지 검토 및 수정 반영 | 브릿지 | 근거 확인, 값 수정, 보완 판단, 단말 반영 승인 |
| G6 | 대사/검증 및 주주명부/KYC/담보 검토 | 브릿지 확장 또는 Agent 보조 | 교차 검증, 담보 서류 검증, 보완 필요성 확정 |
| G7 | 단말 반영 결과 확인 | 단말 | 브릿지 승인 결과를 단말에서 확인 |
| G8 | 심사 요청 또는 보완 요청 분기 | 단말 | 최종 심사 요청 또는 보완 요청 전송 |

## 4. 권장 시연 흐름

### 4.1 정상 흐름
1. G1 신청정보 입력
2. G2 첨부서류 등록 및 AI 분석 요청
3. G3 Agent 처리
4. G4 브릿지 검토 요청 알림
5. G5 브릿지 검토 및 수정 반영
6. G6 담보/실소유자 검증 확인

7. G8 심사 요청

### 4.2 보완 흐름
1. G1 신청정보 입력
2. G2 첨부서류 등록 및 AI 분석 요청
3. G3 Agent 처리
4. G4 브릿지 검토 요청 알림
5. G5 브릿지 검토 및 보완 판단
6. G6 대사/검증 및 주주명부/KYC 확인
7. G7 단말 반영 결과 확인

## 5. 기존 구조 대비 주요 변경점
- 단말 화면에서 "AI 시작"이나 "AI 기능 직접 사용"처럼 보이는 표현은 제거한다.
- 단말 화면에서 AI 제언이나 AI가 서류를 판단/산출하는 표현도 제거한다.
- 단말 화면의 액션은 "AI 분석 요청" 또는 "검토 요청 확인" 수준으로 제한한다.
- Agent 화면은 단말의 확장 화면이 아니라 별도 백그라운드 처리 채널로 보이게 구성한다.
- G3는 실제 사용자가 조작하는 업무 화면이 아니라 시연용 모니터 화면으로 정의한다.
- 브릿지는 단순 확인 화면이 아니라 사람의 판단과 책임이 존재하는 통제 화면으로 강화한다.
- 결과 반영은 브릿지 종료 후 단말로 되돌아오는 구조를 명확히 보여 준다.
- 담보 유형은 일반값이 아니라 `부동산 담보`를 대표 기준으로 고정하고, 담보 관련 문서와 프로세스를 화면 구조에 포함한다.
- 담보평가/권리관계/담보가치 관련 AI 제언을 G3, G5, G6에 분산 배치한다.

1. G1 신청정보 입력

3. G3 Agent 처리 현황
4. G4 브릿지 검토 요청 알림
5. G5 브릿지 검토 및 수정 반영
6. G7 단말 반영 결과 확인
7. G8 심사 요청 또는 보완 요청 분기
8. G6 대사/검증 및 주주명부/KYC 확인 고도화

---

## 7. 결론
현재 목표 화면 체계는 유지 가능하지만, 화면 간 관계는 재정의가 필요하다.
이후 모든 상세 설계와 구현은 다음 구조를 기준으로 해야 한다.

`영업점 단말 입력/요청 -> Agent 자동 처리 -> 브릿지 알림 및 검토 -> 단말 반영 -> 심사/보완 분기`
