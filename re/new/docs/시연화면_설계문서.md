# 시연화면 설계문서

## 분석

### 파일 분석 영역

#### 목적

본 문서는 시연 스텝(S1~S15, 옵션 포함)을 기준으로, 실제 은행 현장에서 기업여신 담당자가 어떤 업무를 수행하는지 정리한 실무 관점 설계 문서다.  
화면 기능 설명이 아니라, "누가/무엇을/어떤 통제로 처리하는지"에 초점을 둔다.

#### 전체 스텝

- S1: To-Be 전체 맵(개요)
- S2: 신청정보 입력(사전조건)
- S3: 첨부문서 등록(2207)
- S4: OCR 문서·필드 추출(AG-01)
- S5: 대사/검증 결과(A2)
- S6: 전결권 상향 검증(A3)
- S7: 진위검증(A4+RPA)
- S8: 심사정보 자동입력(A8)
- S9: 검증결과 확인/수정(1차, 영업점)
- S10(옵): 담보서류 검증(좌표 하이라이트)
- S11(옵): 주주명부 자동입력(CI101)
- S12: TO-DO LIST 선택(CO101)
- S13: 자동입력값 조회(신규)
- S14: 접수 등록 + 전결권 선택/등록
- S15: 반송 확인(영업점 피드백 루프)

#### 프로세스별 설명

### S1. To-Be 전체 맵(개요)
- 영업점 RM/여신 담당자, 심사역, 결재권자가 전체 프로세스와 책임 경계를 먼저 맞춘다.
- 실제 운영에서는 "접수-심사-승인-사후관리"로 역할이 분리되며, 각 단계별 필수 증빙과 승인권자를 사전에 확인한다.
- 실무 포인트: 누락 없는 체크리스트, 단계별 책임자 지정, 반송 기준의 사전 합의.
- 일반인 이해용 설명: 병원에서 진료 전 접수-진료-수납 순서를 먼저 안내받는 것처럼, 은행도 "누가 언제 무엇을 확인하는지"를 먼저 정해야 중간에 책임 공백이 생기지 않는다.

### S2. 신청정보 입력(사전조건)
- 영업점 담당자가 차주 기본정보, 자금용도, 신청금액, 만기, 담보 유형 등을 코어 여신 시스템에 입력한다.
- 차주 식별/고객확인(KYC), 기존 여신 현황, 내부 거래 이력, 기본 적합성 항목을 함께 점검한다.
- 실무 포인트: 신청정보 오입력은 후속 심사/전결 판단 오류로 연결되므로 입력 품질 검증이 핵심이다.
- 일반인 이해용 설명: 온라인 쇼핑 주문서 주소/연락처가 틀리면 배송이 꼬이는 것과 같다. 대출도 시작 정보가 틀리면 뒤 단계(심사, 승인)가 모두 잘못된 전제를 쓰게 된다.

### S3. 첨부문서 등록(2207)
- 영업점 담당자가 사업자등록증, 법인등기부, 재무제표, 세무서류, 계약서, 담보 관련 서류를 스캔/업로드한다.
- 서류별 필수/선택 구분을 확인하고, 누락 시 즉시 보완 요청한다.
- 실무 포인트: 실제 은행은 서류 원본성/최신성(발급일, 유효기간, 직인 여부)을 1차로 확인한다.
- 일반인 이해용 설명: 대출 심사에 필요한 "증거 파일"를 모아 제출하는 단계다. 서류가 오래됐거나 누락되면, 심사 자체를 시작하기 어렵기 때문에 가장 먼저 서류 완전성을 맞춘다.

### S4. OCR 문서·필드 추출(AG-01)
- 기존에는 담당자가 서류를 직접 읽어 주요 필드를 수기로 옮겼고, 자동화 환경에서는 OCR 결과를 검토/보정하는 방식으로 바뀐다.
- 영업점 담당자는 OCR 추출 필드의 신뢰도 낮은 항목을 우선 확인하고, 오인식 항목을 교정한다.
- 실무 포인트: OCR은 "입력 보조"이며 최종 책임은 담당자 확인(HITL)에 있다.
- 일반인 이해용 설명: 종이 문서를 사진으로 찍어 글자를 자동으로 읽는 기능이다. 다만 기계가 100% 정확하지 않아서, 사람이 마지막으로 맞는지 확인하는 절차가 꼭 필요하다.

### S5. 대사/검증 결과(A2)
- 담당자가 전산 입력값과 제출서류 추출값의 불일치, 누락, 형식 오류를 항목 단위로 확인한다.
- 오류 유형별로 조치한다: 즉시 정정 가능 항목은 수정, 증빙 불충분 항목은 보완 요청, 판단 애매 항목은 심사역 협의.
- 실무 포인트: 수정 이력/사유를 남겨 감사 추적이 가능해야 한다.
- 일반인 이해용 설명: "신청서에 적은 값"과 "실제 증빙서류 값"을 대조표로 맞춰보는 과정이다. 값이 다르면 왜 수정했는지 기록을 남겨야 나중에 문제가 생겼을 때 추적할 수 있다.

### S6. 전결권 상향 검증(A3)
- 심사역/여신 담당자가 특약, 조건부 조항, 리스크 문구를 확인해 전결권 레벨(영업점/본부/상향)을 판단한다.
- 상향 사유 발생 시 근거 문구와 함께 상신 경로를 변경하고, 결재권자 승인 체계에 맞춰 재분류한다.
- 실무 포인트: 전결권 오판은 내부통제 이슈로 직결되므로 근거 기반 판단이 필수다.
- 일반인 이해용 설명: 금액이 크거나 조건이 복잡한 건은 더 높은 직급의 승인이 필요하다. 이 단계는 "누가 최종 결재해야 하는지"를 규정에 맞게 정하는 안전장치다.

### S7. 진위검증(A4+RPA)
- 담당자가 발급기관 확인이 필요한 서류를 선별하고, 발급번호/식별값으로 진위 조회를 수행한다.
- 자동화 환경에서는 RPA가 조회를 수행하고, 담당자는 결과(정상/비정상/미확인)와 제출본-발급본 대사 결과를 검토한다.
- 실무 포인트: 미확인/불일치 건은 대출 진행 보류 또는 보완 요구 대상이다.
- 일반인 이해용 설명: 제출된 서류가 진짜인지 "원 발급기관 기준"으로 다시 확인하는 단계다. 위조 가능성이 있으면 바로 통과시키지 않고 멈추고 재확인한다.

### S8. 심사정보 자동입력(A8)
- 검증 완료된 항목만 심사 전자서식으로 반영하고, 예외/저신뢰 항목은 미반영 상태로 남겨 담당 확인을 요구한다.
- 심사역은 자동입력 결과를 검토하고 필요한 보정 후 저장한다.
- 실무 포인트: 자동입력 범위와 예외 항목이 명확히 분리되어야 책임 소재가 분명해진다.
- 일반인 이해용 설명: 맞다고 확인된 정보만 자동으로 심사 화면에 채워 넣는다. 애매한 정보까지 자동으로 넣지 않는 이유는 실수 확산을 막기 위해서다.

### S9. 검증결과 확인/수정(1차, 영업점)
- 영업점 담당자가 1차 확정자로서 이슈 항목을 재확인하고 최종 수정 또는 예외 사유를 입력한다.
- 모든 확인필요 항목 해소 후 "확정" 처리하여 심사부 단계로 넘긴다.
- 실무 포인트: 현업 확정 단계가 있어야 심사부에서 재작업이 줄어든다.
- 일반인 이해용 설명: 현장에서 고객을 직접 응대한 담당자가 마지막으로 한 번 더 확인하는 단계다. 여기서 정리가 잘 되어야 본부 심사에서 되돌아오는 일이 줄어든다.

### S10(옵션). 담보서류 검증(좌표 하이라이트)
- 담보 담당자/심사역이 등기부, 대장, 감정/사정 의견서 등에서 핵심 근거를 대조한다.
- 소재지, 소유자, 면적, 권리관계, 담보가액 산정 근거를 신청정보와 비교한다.
- 실무 포인트: 담보 불일치 건은 한도/조건 조정 또는 추가 담보 요구로 이어진다.
- 일반인 이해용 설명: 담보로 잡는 부동산/자산 정보가 신청 내용과 실제 문서가 같은지 확인한다. 다르면 대출 한도 축소나 추가 담보 요청이 발생할 수 있다.

### S11(옵션). 주주명부 자동입력(CI101)
- 담당자가 주주명부를 확인해 지분율 합계, 최대주주, 동일인/특수관계인 여부를 점검한다.
- 자동화 환경에서는 OCR 구조화 후 CI101에 반영하고, 담당자가 최종 확정한다.
- 실무 포인트: 실질 지배구조 확인은 여신 심사와 내부통제(이해상충/편중) 관점에서 중요하다.
- 일반인 이해용 설명: 회사의 실제 주인이 누구인지, 특정 사람에게 지분이 과도하게 몰려 있는지 확인하는 과정이다. 이는 대출 리스크와 직결되는 기본 정보다.

### S12. TO-DO LIST 선택(CO101)
- 본부 심사역이 접수 대기 목록에서 우선순위(자동입력 완료, 확인필요, 반송 이력 등)를 기준으로 건을 선택한다.
- 선택 즉시 핵심 요약(차주, 신청금액, 리스크 플래그, 미해소 이슈)을 확인하고 심사 착수 여부를 결정한다.
- 실무 포인트: TO-DO 선별 규칙이 명확할수록 처리 리드타임이 안정된다.
- 일반인 이해용 설명: 병원 응급실의 트리아지처럼, 심사도 먼저 처리할 건을 고르는 단계가 필요하다. 우선순위를 잘 잡아야 전체 처리 속도가 좋아진다.

### S13. 자동입력값 조회(신규)
- 심사역이 자동입력값, 검증결과, 원문 근거를 한 화면에서 교차 검토한다.
- 불일치/확인필요 항목은 추가 확인 요청 또는 직접 보정 후 다음 단계로 넘긴다.
- 실무 포인트: 근거와 값을 분리 조회하면 검토 시간이 길어지므로 통합 뷰가 효과적이다.
- 일반인 이해용 설명: "결과값"과 "그 결과의 근거"를 동시에 보는 검토 단계다. 숫자만 보면 놓칠 수 있는 오류를 원문 근거까지 같이 보며 줄인다.

### S14. 접수 등록 + 전결권 선택/등록
- 심사역이 심사 접수 등록을 수행하고, 전결권 체계에 맞춰 결재 경로를 지정한다.
- 상향 필요 건은 상향 전결로 등록하고, 일반 건은 통상 전결로 등록해 승인 프로세스를 개시한다.
- 실무 포인트: 접수 시점의 전결권 등록 정확도가 승인 지연/재상신 여부를 좌우한다.
- 일반인 이해용 설명: 이제 심사 건을 공식 결재 라인에 태우는 단계다. 결재 라인을 잘못 태우면 다시 올려야 해서 승인 일정이 늦어진다.

### S15. 반송 확인(영업점 피드백 루프)
- 반송 건은 영업점이 사유(서류 미비, 불일치, 진위 미확인, 전결 근거 부족)를 확인하고 보완 계획을 수립한다.
- 보완 서류 재업로드 후 재검증/재심사로 재진입하며, 동일 사유 반복 여부를 관리한다.
- 실무 포인트: 반송 사유 코드화와 재발 방지 관리가 품질 개선의 핵심이다.
- 일반인 이해용 설명: 한 번에 승인되지 않은 건을 왜 반려됐는지 분석하고 다시 제출하는 단계다. 중요한 것은 "다음에 같은 이유로 다시 반려되지 않게" 원인을 관리하는 것이다.

#### 공통 실무 원칙(HITL/내부통제)

- 자동화 결과는 참고/보조 수단이며, 최종 판단과 확정 책임은 은행 직원에게 있다.
- 불일치/저신뢰/진위 미확인 항목은 예외 사유와 조치 이력을 남겨야 한다.
- 전결권, 승인, 반송은 모두 근거 문서 기반으로 판단하고 이력 추적 가능해야 한다.

#### 외부 확인 근거(요약)

- 은행 기업대출 안내 페이지의 제출서류 항목(법인등기부, 이사회 의사록, 재무자료 등).
- 은행 기업여신 약정서/약관의 자료제출, 신용상태 악화 시 감액·정지, 서류 허위·위변조 관련 조항.
- 기업뱅킹 업무 메뉴의 대출상담/약정/진행현황/서류제출/상환 등 실제 운영 기능 구성.

### 직장인 신용대출 기준 프로세스 영역

#### 직장인 신용대출 기준 프로세스

- 1) 상담/한도 가조회
  - 구분: 오프라인 + 시스템
  - 사용 화면/시스템: `한도 가조회 화면` (모바일뱅킹 앱, 영업점 여신상담 단말)
  - 은행원 업무 수행 방식: 고객의 자금 목적·희망금액·상환기간을 먼저 듣고, 시스템 가조회 결과와 비교해 가능한 범위와 한계(금리/한도)를 설명한다.
- 2) 본인확인/고객확인(KYC)
  - 구분: 시스템 중심 (필요 시 오프라인 보완)
  - 사용 화면/시스템: `본인확인/KYC 화면` (CIF, AML/FDS)
  - 은행원 업무 수행 방식: 실명확인, 고객정보 최신화, 이상거래 징후 점검을 수행하고 의심 신호가 있으면 추가 확인 절차를 진행한다.
- 3) 재직·소득 확인
  - 구분: 시스템 + 오프라인 보완
  - 사용 화면/시스템: `재직·소득 검증 화면` (공공데이터 연계 모듈, 증빙서류 등록 화면)
  - 은행원 업무 수행 방식: 자동 연계 데이터와 제출서류를 대조해 재직 안정성과 소득 일관성을 확인하며, 불일치 항목은 추가 증빙을 받아 정정한다.
- 4) 신용평가/부채점검
  - 구분: 시스템
  - 사용 화면/시스템: `신용평가 결과 화면` (CSS, 부채현황 조회)
  - 은행원 업무 수행 방식: 내부/외부 신용지표와 총부채 수준을 점검해 상환부담 가능 여부를 판단하고, 과다 부채 시 한도·조건 조정 필요성을 검토한다.
- 5) 조건 산출
  - 구분: 시스템 + 오프라인 설명
  - 사용 화면/시스템: `조건 산출 화면` (여신심사 단말)
  - 은행원 업무 수행 방식: 시스템이 제시한 한도·금리·만기·상환방식 후보를 검토해 실행 가능한 안을 정리하고 고객에게 선택지별 차이를 설명한다.
- 6) 예외 심사(필요시)
  - 구분: 오프라인 협의 + 시스템 상신
  - 사용 화면/시스템: `예외심사 상신 화면` (심사포털 워크플로우)
  - 은행원 업무 수행 방식: 자동심사 기준 밖의 건은 사유와 근거서류를 첨부해 수동 심사로 넘기고, 보완 요청·재판정 결과를 이력으로 남긴다.
- 7) 승인/전결
  - 구분: 시스템
  - 사용 화면/시스템: `승인/전결 화면` (전자결재시스템)
  - 은행원 업무 수행 방식: 권한 체계에 맞춰 결재선을 지정하고 승인 요청을 진행하며, 상향 전결이 필요하면 근거를 보강해 재상신한다.
- 8) 약정/설명
  - 구분: 오프라인 설명 + 시스템 약정
  - 사용 화면/시스템: `약정/설명 확인 화면` (전자약정시스템)
  - 은행원 업무 수행 방식: 금리·연체이자·중도상환수수료·기한이익상실 조건을 설명한 뒤 전자서명을 받아 약정을 체결하고 약정서 교부를 안내한다.
- 9) 실행
  - 구분: 시스템 + 오프라인 재확인
  - 사용 화면/시스템: `실행 화면` (코어뱅킹 여신실행 단말)
  - 은행원 업무 수행 방식: 입금 계좌와 실행 조건을 최종 확인한 뒤 대출금을 실행하고, 첫 납입일·상환 스케줄·자동이체 설정을 고객과 다시 확인한다.
- 10) 사후관리
  - 구분: 시스템 모니터링 + 오프라인 상담
  - 사용 화면/시스템: `사후관리 대시보드` (EWS/연체관리 시스템)
  - 은행원 업무 수행 방식: 연체 징후를 상시 모니터링해 선제 안내를 수행하고, 금리인하·만기연장·재약정 요청 발생 시 내부 기준에 따라 재심사한다.

## 신규 설계

### 제목 : 기업여신 신청(서류입력 및 검수 자동화)

### 전체 스텝(작성중)

- P1. 신청정보 입력 : 단말
- P2. 첨부문서 등록 : 단말
- P3. 실행 계획 수립 : Agent
- P4. 문서 전처리 : Agent
- P5. 진위검증 : Agent
- P6. OCR 및 필드 추출 : Agent
- P7. 오류 알림 : 단말 + 브릿지 미니
- P8. 값 수정(HITL) : 브릿지
- P9. 대사/검증 : Agent
- P10. 완료 알림 : 단말 + 브릿지 미니
- P11. 결과 확인(HITL) : 브릿지
- P12. 심사정보 자동입력 및 심사 요청 : 단말

### 프로세스별 설명

- P1. 신청정보 입력
  - 화면: 단말
  - 대출신청 기본정보 입력 화면
  - 시연 화면에는 `법인명`, `승인신청번호`, `여신과목`, `신청금액`, `여신기간(만기일)`, `자금용도`, `상환방식`, `담보유형`만 표시한다.

- P2. 첨부문서 등록
  - 화면: 단말
  - 제출서류를 업로드한다.
  - 시연 업로드 파일(예시):
    - `사업자등록증명원.pdf`
    - `법인등기부등본.pdf`
    - `표준재무제표증명.pdf`
    - `부가가치세 과세표준증명원.pdf`
    - `소득금액증명원.pdf`
  
- P3. 실행 계획 수립
  - 화면: Agent
  - 에이전트가 처리 순서와 우선순위를 정하고 업로드된 문서를 분석해 `문서 전처리 대상`, `OCR 수행 대상`, `진위검증 대상`, `대사/검증 대상`으로 분류한다.
  
- P4. 문서 전처리
  - 화면: Agent
  - OCR 정확도 향상을 위해 에이전트가 전처리 Tool을 호출해 문서 기울기 보정, 노이즈 제거, 명암/대비 보정, 페이지 정렬을 수행한다.

- P5. 진위검증
  - 화면: Agent
  - 발급기관 확인 대상 문서를 선별해 진위 결과를 조회/확인한다.
  - 미확인/불일치 결과는 보류 사유로 기록하고 추가 확인 절차를 진행한다.

- P6. OCR 및 필드 추출
  - 화면: Agent
  - 자동 추출된 필드 중 신뢰도 낮은 항목을 우선 검토하고 오인식 값을 교정한다.
  - 추출값과 원문 근거의 연결 상태를 확인해 후속 대사 단계 품질을 확보한다.

- P7. 오류 알림
  - 화면: 단말 + 브릿지 미니
  - Agent가 OCR 및 필드 추출 중 오류발생한 사항을 브릿지 미니를 통해 알림

- P8. 값 수정(HITL)
  - 화면: 브릿지
  - 담당자가 원문 근거를 확인한 뒤 값 수정 또는 예외 유지 중 하나를 결정한다.
  - 수정/예외 사유를 반드시 기록해 감사 추적이 가능하도록 남긴다.

- P9. 대사/검증
  - 화면: Agent
  - 전산 입력값과 추출값을 항목별로 비교해 불일치, 누락, 형식 오류를 확정한다.
  - p9의 기본 레이아웃은 p6과 같음. 대사/검증에서는 입력값 대사, 문서간 대사, 필수값 누락 검증, 타입 검증, 유효값 검증, KYC 교차검증 등을수행한다.

- P10. 완료 알림
  - 화면: 단말 + 브릿지 미니
  - Agent가 실행 완료한 사항을 브릿지 미니를 통해 알림

- P11. 결과 확인(HITL)
  - 화면: 브릿지
  - 담당자가 최종 결과(정상/예외/보완필요)를 확인하고 1차 확정 여부를 결정한다.
  - 확정 불가 건은 반송 사유를 명시해 재제출 루프로 넘긴다.

- P12. 심사정보 자동입력 및 심사 요청
  - 화면: 단말
  - 확정된 항목을 심사 화면에 반영하고 누락/예외 항목을 심사 메모로 전달한다.
  - 심사 요청을 등록하고 전결선 기준에 맞춰 본부 심사 단계로 이관한다.

### 화면 템플릿 설계

#### 템플릿 유형 정의

- 단말
  - 정의: 은행 직원이 실제 업무를 수행하는 통합단말 화면 템플릿.
  - 용도: 입력, 조회, 확정, 심사 요청 등 실제 사용자 액션 중심 화면.
- Agent
  - 정의: AI 에이전트 내부 처리 상태를 시각화하는 운영 모니터형 화면 템플릿.
  - 용도: OCR 진행, 검증 단계, 진위조회, 자동입력 실행 등 "AI가 지금 무엇을 하는지" 시연용으로 표현.
  - 비고: 실제 최종 사용자가 일상적으로 보는 업무 화면은 아님.
- 브릿지
  - 정의: AI 수행 결과를 사람이 확인/수정/승인할 수 있도록 전달하는 중간 확인 화면 템플릿.
  - 용도: 불일치 목록, 근거 하이라이트, HITL 수정, 예외 사유 입력.
- 브릿지 미니(+@)
  - 정의: 메신저 신규 메시지처럼 우측 하단에 뜨는 소형 알림 UI 템플릿.
  - 용도: 오류 발생, 검증 완료, 확인 요청 등 이벤트성 알림 전달.

#### 공통 화면 크기 기준(캡처/녹화/PPT 반영)

- 기본 출력 비율: 16:9 고정.
- 메인 화면(단말/Agent/브릿지) 공통 해상도: HD 1280x720(px).
- 녹화 및 캡처는 1280x720 기준으로 제작해 PPT 삽입 시 화면 왜곡이 없도록 한다.
- 브릿지 미니는 단독 페이지를 별도로 만들지 않고, 1280x720 메인 화면 캔버스 안 우측 하단 오버레이 컴포넌트로 구현한다.
- 모든 HTML은 동일한 외곽 프레임(헤더 높이, 본문 영역, 여백 체계)을 사용해 화면 전환 시 크기/구도가 흔들리지 않도록 한다.

#### HTML 템플릿 공통 규칙

- 파일 단위: 화면 1개 = HTML 1개.
- 공통 래퍼: `app-shell`(고정 1280x720) 내부에 `header`, `content`, `footer(optional)` 구조를 사용한다.
- 모든 화면은 공통적으로 상단 `Title` 영역을 반드시 포함한다.
- 폰트/색상/간격 토큰은 공통 CSS 변수로 관리해 템플릿 간 일관성을 유지한다.
- 캡처 안정성을 위해 반응형 분기 대신 고정 캔버스 레이아웃을 우선 적용한다.
- 애니메이션은 녹화 시 가독성을 위해 0.2~0.8초 범위의 짧은 트랜지션만 사용한다.

#### 템플릿별 상세 표현 원칙

- 단말 화면
  - 실제 은행 통합단말 UI와 최대한 유사한 톤(정보 배치, 입력 필드, 조회/저장 버튼 흐름)으로 구성한다.
  - 다만 시연용 특성을 반영해 과도한 데이터 행/컬럼은 제거하고, 핵심 업무 데이터만 최소 표시한다.
  - 글씨가 지나치게 작아지지 않도록 가독성 우선으로 설계하며, 주요 값/상태/버튼은 한눈에 식별 가능해야 한다.

- Agent 화면
  - 화면 상단에는 에이전트가 수행하는 전체 프로세스 흐름(예: 단계 바/스텝 트래커)을 고정으로 노출한다.
  - 현재 실행 중인 프로세스를 상단 흐름에서 명확히 하이라이트한다.
  - 화면 하단에는 현재 단계의 진행 과정을 시각적으로 표시한다(진행률, 로그, 상태 변화, 작업 결과 등).

- 브릿지 화면
  - 좌측 패널에는 파일 목록(첨부 문서/처리 대상)을 배치한다.
  - 메인 영역에는 에이전트 수행 결과를 집중 표시한다(대사 결과, 근거 하이라이트, 확인 필요 항목, HITL 조치).
  - 사용자의 핵심 동선은 `파일 선택 -> 결과 확인 -> 수정/확정` 흐름으로 단순하게 유지한다.
